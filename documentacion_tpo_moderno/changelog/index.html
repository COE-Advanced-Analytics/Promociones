📋 Changelog v3.0 — Desarrollo Mar-2026

Este changelog documenta todos los cambios implementados en TPO v3.0 respecto a v2.0, desarrollados durante marzo 2026. Los cambios están agrupados por área funcional.

GRUPO 1 — Arquitectura Core

1. Horizonte M+2 con weeks_ahead configurable run_batch_optimizer.py

En v2.0 el horizonte de predicción era fijo. En v3.0 se calcula dinámicamente desde el CSV de configuración.

# Cálculo dinámico del horizonte
target_date = date(target_year, 1, 1) + timedelta(weeks=target_week - 1)
last_date   = date(data_year,   1, 1) + timedelta(weeks=data_week   - 1)
WEEKS_AHEAD = (target_date - last_date).days // 7

# Configuración activa (feb 2026)
# datos hasta: year=2026 week=9 | target: year=2026 week=17 → weeks_ahead=8

El modelo de elasticidad avanza weeks_ahead pasos en silencio antes de guardar las predicciones, garantizando que las estimaciones correspondan a la semana objetivo real (M+2).

3. Universo de SKUs y precios desde Grow (precio_efectivo) data_factory.py + query_precio_efectivo.sql

En v2.0 el universo de SKUs activos y el precio_base venían de mapa_de_precios (tabla interna Alicorp). En v3.0 vienen de Grow (gld_cliente.grow_cliente_detalle), que refleja ventas reales en canal moderno.

-- query_precio_efectivo.sql — fuente Grow
SELECT
  EXTRACT(ISOYEAR FROM fec_venta) AS year,
  EXTRACT(ISOWEEK  FROM fec_venta) AS week,
  id_material_origen               AS cod_material,
  SUM(mnt_venta_sin_igv)           AS Sum_mnt_bruto,
  SUM(cnt_material)                AS Sum_cnt_material_venta,
  SAFE_DIVIDE(SUM(mnt_venta_sin_igv), SUM(cnt_material)) AS p_u
FROM `acpe-prod-dlk-shrd-gld.gld_cliente.grow_cliente_detalle`
WHERE periodo >= '2023-01-01'
  AND des_canal = 'MODERN'
  AND des_cadena = '{subcadena}'
GROUP BY ALL
ORDER BY year, week, cod_material

Historia disponible: desde 2023-01-01 → ~164 semanas. El precio_base se deriva como el precio máximo reciente observado por SKU×subcadena en este dataset.

GRUPO 2 — Fórmula ROI → GTN Nueva Propuesta (feb 2026)

4. Nueva fórmula GTN en UCOs optimizer_model.py · _find_optimal_window

En v2.0 el uplift se calculaba en kilos mezclados con el BI unitario en S/ por UCO, produciendo ROI negativos incorrectos. En v3.0 todo se calcula en UCOs (Unidades de Consumo).

Fórmula GTN Nueva Propuesta

Inversión(precio_base − precio_promo) × un_promo
Venta Brutaprecio_base × un_promo
Costoscogs_unitario × un_promo
BI TotalVenta Bruta − Inversión − Costos
BI UnitarioBI Total / un_promo
Uplift (UN)un_promo − un_base
BI UpliftBI Unitario × Uplift (UN)
ROI(BI Uplift / Inversión) − 1

Guards: se descarta el escenario si uplift_units ≤ 0 OR bi_uplift ≤ 0 OR inversion ≤ 0

# Conversión kilos → UCOs
un_promo = int(np.floor(qty_promo / max_kilos_unidad))
un_base  = int(np.floor(qty_base  / max_kilos_unidad))

inversion    = (precio_base - precio) * un_promo
venta_bruta  = precio_base * un_promo
costos       = cogs_unitario * un_promo
bi_total     = venta_bruta - inversion - costos
bi_unitario  = bi_total / un_promo
uplift_units = un_promo - un_base
bi_uplift    = bi_unitario * uplift_units
roi          = round((bi_uplift / inversion - 1) * 100, 2)

if uplift_units <= 0 or bi_uplift <= 0 or inversion <= 0:
    continue

GRUPO 3 — Predicción y Distribución Diaria

6. Perfil diario ponderado real por SKU×subcadena optimizer_model.py · _build_perfil_dict

En v2.0 la demanda semanal se distribuía uniformemente entre los 7 días (1/7 por día). En v3.0 se usa el perfil real de ventas por día de semana para cada SKU×subcadena, calculado desde las últimas 8 semanas de demanda_ajustada.

# Estructura del perfil diario
_perfil_dict: Dict[(cod_material, des_subcadena), Dict[dia_semana, pct]]
# dia_semana: 0=Lunes, 1=Martes, ..., 6=Domingo
# pct: proporción de la demanda semanal que cae ese día (suma=1.0)

# Fallback si no hay perfil para el SKU
perfil_uniforme = {d: 1/7 for d in range(7)}

7. Evaluación exhaustiva de todas las ventanas diarias optimizer_model.py · _find_optimal_window

Para cada precio candidato se evalúan 98 combinaciones de ventana: 7 días de inicio × 14 duraciones posibles. Se selecciona la combinación con ROI máximo.

for start_day in range(7):       # 0=Lun … 6=Dom
    for duration in range(1, 15): # 1 a 14 días
        indices   = list(range(start_day, min(start_day + duration, 28)))
        qty_promo = sum(daily_promo[i] for i in indices)
        qty_base  = sum(daily_base[i]  for i in indices)

# Cálculo de fechas concretas por escenario

GRUPO 5 — Filtros del Reporte Final (_9_)

10. Filtros de negocio en _get_best_scenario_by_roi optimizer_model.py

Se aplican 4 filtros en orden antes de seleccionar el mejor escenario por SKU:

#FiltroCriterioRazón
1cantidad_predicha≥ 5Volumen mínimo operativo
2duracion_dias< 16Máximo 15 días de promo
3factor descuento6% ≤ factor ≤ 20%Regla dura de negocio — descuentos fuera de rango no se usan
4elasticidad< 0Solo SKUs donde bajar precio sube la demanda (economicamente correcto)
# Fórmula del factor de descuento
factor = round((precio_base - precio_promo) / precio_base, 2)
# Ejemplo: precio_base=10, precio_promo=8.5 → factor=0.15 (15%)

# Fórmula de elasticidad precio-demanda
elasticidad = ((qty_promo - qty_base) / qty_base) / ((precio - precio_base) / precio_base)
# Valor esperado: negativo (precio baja → demanda sube)

GRUPO 6 — Fixes Técnicos (Mar-2026)

11. Fix cast cod_material en cog_unitario data_factory.py

El bloque de cog_unitario no tenía cast a Int64, a diferencia de todas las otras tablas. Esto causaba que el lookup fallara por tipo string vs Int64, excluyendo SKUs que sí tenían COGS.

# FIX — agregar cast igual que todas las demás tablas
df['cod_material'] = pd.to_numeric(df['cod_material'], errors='coerce').astype('Int64')
# Impacto: 429 → 570 filas de COGS disponibles

12. Fix float comparison para precio_base optimizer_model.py

La comparación precio == precio_base fallaba por precisión de punto flotante. Se reemplazó con np.isclose.

# Antes — fallaba por float precision
mask = grupo['precio'] == precio_base_sku

# Fix — tolerancia ±0.01%
mask = np.isclose(grupo['precio'].astype(float), float(precio_base_sku), rtol=1e-4)

13. Fix KeyError cod_material_sku data_factory.py

promocion_detalle se procesaba antes que precio_efectivo, por lo que cod_material_sku aún no existía. Fix: guardar como _promocion_detalle_raw durante el loop y procesar al final.

14. Fix ValueError NA to integer optimizer_model.py

# Fix — dropna antes del cast
df_unidades = df_unidades.dropna(subset=['cod_material'])
df_unidades['cod_material'] = df_unidades['cod_material'].astype(int)

15. Query cog_unitario — SAFE_CAST + limpieza alfanuméricos query_cogs_unitario.sql

La query original fallaba con Bad int64 value: DS952V al encontrar materiales alfanuméricos. Se reemplazó CAST por SAFE_CAST con LTRIM de ceros y filtro de NULLs.

-- Antes
CAST(cod_material AS INT64)

-- Fix
SAFE_CAST(LTRIM(CAST(cod_material AS STRING), '0') AS INT64)
-- + WHERE cod_material IS NOT NULL para descartar alfanuméricos
